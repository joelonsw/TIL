## FastAPI 구조
- **개요**
  - 파이썬의 Type Hint를 극한으로 활용하는 프레임워크
  - 가볍고 빠르며, 필요한 컴포넌트 조립해서 쓰는 형태
  - Pydantic + Type Hints
  - Event Loop + Threadpool 로 구성

- **요청 처리 라이프사이클**
  - Starlette(웹 라우팅 툴킷) + Pydantic(데이터 검증 라이브러리) 조합
  - 흐름도
    1. Client Request
    2. ASGI Server: Play Netty같은 역할. 
       - Asynchronous Server Gateway Interface (Uvicorn)
       - 들어온 HTTP 요청을 파이썬이 이해할 수 있는 이벤트로 변환하여 비동기 이벤트 루프에 올림
    3. FastAPI(Starlette) Router: 요청 URL, HTTP 메서드 분석해 적절한 엔드포인트로 라우팅
    4. Pydantic Validation: 엔드포인트 도달 전, Type Hint 기반 요청 데이터 파싱/유효성 검사
    5. DI: 비즈니스 로직 실행 전 필요한 의존성 주입
    6. Endpoint Execution (Sync, Async)
        - `async def`
          - 메인 이벤트 루프에서 비동기적으로 실행
          - DB I/O나 외부 API 호출 시, `await`만나면 다른 요청 처리하도록 제어권 양보
        - `def`
          - FastAPI가 이를 감지하고 외부 ThreadPool로 던져서 실행
          - 동기 함수 내에서 블로킹 작업 하더라도 메인 이벤트 루프 멈추지 않음

- **핵심 특징**
  1. GIL(Global Interpreter Lock)의 한계
     - JVM은 멀티 스레딩으로 CPU 코어를 십분 활용
     - 하지만, 파이썬은 GIL로 인해 한 프로세스 안에서 동시에 여러 스레드가 파이썬 바이트코드 실행이 불가능
     - 따라서 컨테이너 기반으로 배포시, 하나의 거대 인스턴스를 띄우기 보단, 프로세스 매니저를 사용해 CPU 코어만큼 Uvicorn 워커 프로세스 띄우는 방식으로 스케일 아웃
  2. 심플한 DI
     - `Depends()`라는 함수형 스타일의 DI 활용
     - 싱글톤 빈 관리보다는, "이 요청 처리하기 전에 이 함수를 먼저 실행해서 결과값을 넘겨줘"에 가까움
  3. 동기/비동기의 혼용 주의
     - 비동기 I/O의 이점을 누리기 위해서는 엔드포인트를 `async def`로 선언하고
     - 내부 라이브러리도 모두 비동기용 (ex. `httpx`, `asyncpg`) 사용해야 함
       - async def 내부에서 동기함수 호출하면 메인 이벤트 루프가 블로킹되어 전체 서버가 멈추는 대참사 발생

- **이벤트 루프의 이해**
  - 비동기(`await`)의 마법
    - `await`는 스레드가 대기하는 것이 아님
    - 네트워크 I/O나 타이머를 OS 커널과 물리적 네트워크 장비에 위임. 메인 스레드는 즉시 다음 요청을 받으러 감 
    - 따라서, `async def` 내부에서 동기 함수 (`requests`, `time.sleep`) 호출 시, 1개뿐인 메인 스레드가 잡혀버릴 수 있음 => 먹통
  - 하이브리드 전략: 동기 함수 격리하기
    - 비동기 DB(`asyncpg`), 외부 API(`httpx`)로 전체 로직 도배가 이상적이지만, 레거시 동기 SDK를 써야한다면, 다음과 같이 처리
    - `asyncio.to_thread()`: 무겁고 블로킹되는 동기 작업만 쓰레드풀에 던지기
    - 애초에 라우터를 `def`로 선언하면 FastAPI가 전체를 스레드 풀로 넘김
    - 하지만, 비동기 I/O의 퍼포먼스를 위해 `async def`로 뼈대잡고, 동기 작업은 `to_thread` 형식으로 격리하는 것도 좋음
  - Scala/Play에서 Akka에 보내는것도 같은 맥락
    - 무거운 동기 작업은 격리된 스레드풀로 유배 보내주세요!

## FastAPI vs Spring
- JVM: `Thread-per-request`
  - 요청올 때마다, 쓰레드 하나씩 전담마크
  - 요청A가 OpenAI 응답을 10초 동안 멍하니 기다려도, 요청B는 무난히 처리 가능
- FastAPI: `asyncio` 이벤트 루프 (싱글 스레드)
- FastAPI를 `uvicorn app main:app` 으로 띄우면...
  1. 메인 스레드(이벤트 루프): 1개
     - 역할: 비동기 `async def` 코드를 실행하고, OS로 부터 완료 알림(interrupt)를 받아 콜백을 실행하는 스레드
     - 특징: 이 1개의 스레드가 수천/수만개의 동시접속 처리
  2. 스레드풀: 기본 40개
     - `asyncio.to_thread()`
       - 파이썬 내장 `asyncio`의 디폴트 스레드 풀 사용
       - 갯수: `min(32, OS 코어 수 + 4)`
     - 라우터를 `def`로 생성 (FastAPI 자동 위임)
       - 내부적으로 비동기 호환 라이브러리 `AnyIO` 사용
       - `AnyIO` 디폴트 스레드풀 제한은 코어수와 무관하게 40개

## FastAPI k8s 배포
1. Pod 하나에 CPU Core를 여러개 주었을 때
   - `limits: cpu: "4"`를 주고, `uvicorn main:app`으로 배포했다면 최악
   - 파이썬은 단일 프로세스 -> CPU 1개만 혹사시키고, 나머지 3개의 CPU는 놀게됨
   - 이때는 `Gunicorn`을 매니저로 써서 워커 4개로 띄우던가 해야함
   - 요청 로드밸런싱은 OS단에서 
2. Pod 하나에 CPU 코어 1개만 준 경우
   - `limits: cpu: "1"`를 주고, `uvicorn main:app`으로 배포했다면 최고
   - 어차피 Pod당 CPU 1개밖에 못쓰니, 파이썬 프로세스도 1개만 띄우는 것이 효율적
   - 트래픽이 몰리면 k8s의 hpa가 pod 갯수 늘려주면 됨
   - 작은 Pod를 여러개 띄우자!

## 파이썬 GIL
*참고: https://wikidocs.net/234356*  
*참고: https://www.youtube.com/watch?v=hj8BnSAalEs*  
- **개요**
  - 파이썬에서는 multi-threading 처리시, GIL 때문에 느려진다는 얘기가 있음
  - 다수의 thread가 동시에 python byte code를 실행하지 못하게 막는 mutex의 일종
  - 결과적으로, mutli-thread가 single-thread와 비슷한 성능을 내게 됨

- **GIL**
  - 하나 공통된 자원 violation 메모리 보호 (여러 Thread 들이 접근하는 공통된 자원)
  - GC의 reference count 변수가 이 critical section
  - Cpython 인터프리터와 같은 환경에서 임계 영역을 Mutex 형식으로 보호
  - CPU Bound 연산은 멀티 스레드 방식으로 병렬처리가 어려움 => 파이썬에서는 멀티 프로세스 고려
